# Validation Experiments

Use when a market-validation or product strategy hypothesis needs a concrete
test before becoming product scope, backlog, or implementation work.

This skill designs experiments. It does not run product implementation or
declare product behavior valid without evidence.

## Source Order

1. Latest market-validation report, `hypothesis-scoring` scorecard, pain
   clusters, competitor map, economics model, and user brief.
2. Existing product/spec/scenario/backlog docs and active work lanes.
3. Available channels, analytics, prototype tools, interview access, support
   workflows, sales notes, landing pages, and current product surfaces.
4. `docs/patterns/workflow.md` for the Doc Routing Decision Matrix.

Use live research for current channel availability, competitor response,
regulatory constraints, and geography-specific assumptions.

## Experiment Types

- customer interview;
- problem interview;
- landing page or waitlist test;
- pricing or willingness-to-pay test;
- concierge pilot;
- prototype usability test;
- smoke test;
- workflow simulation;
- manual service trial;
- data or integration feasibility probe.

## Checklist

1. Define the hypothesis and the riskiest assumption.
2. Choose the cheapest experiment that can change the decision.
3. Define target participant, channel, script or task, data needed, and sample
   size.
4. Define success, kill, inconclusive, and follow-up criteria before running.
5. Identify product, design, brand, operations, legal, or data constraints.
6. Route implementation-heavy experiments to `plan-change` only after the
   product behavior and acceptance criteria are clear.
7. Route evidence-backed product/spec handoff to `synthesis-to-spec`.
8. Route durable backlog candidates to `docs/backlog/_index.md` with
   acceptance criteria.
9. Record Doc Routing Decision Matrix rows for experiments created, deferred,
   blocked, or intentionally not documented.

## Output

- experiment plan and risk tested;
- success, kill, and inconclusive criteria;
- participant or channel plan;
- required artifacts and evidence;
- backlog or spec route;
- Doc Routing Decision Matrix rows.
