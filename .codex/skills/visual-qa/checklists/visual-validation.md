# Visual Validation Checklist

- [ ] Expected source named.
- [ ] Product/spec source checked when visible behavior, copy, scenario state,
      or task completion is judged.
- [ ] UX/layout source checked when hierarchy, density, carried state, or
      interaction flow is judged.
- [ ] Brand source checked when naming, tone, trust language, message
      hierarchy, or visual direction is judged.
- [ ] Design-system source checked when tokens, components, layout rules,
      states, or visual evidence expectations are judged.
- [ ] Desktop and mobile/narrow viewports included or explicitly skipped.
- [ ] Long content and dynamic states considered.
- [ ] Text overlap, clipping, and horizontal overflow checked.
- [ ] Primary action and scan hierarchy checked.
- [ ] Token/component consistency checked.
- [ ] Feature-specific UX findings are routed to `ux-flow-review` and the
      owning product/spec docs.
- [ ] Reusable design/layout/token/component findings are routed to
      `design-system` and `docs/design/`.
- [ ] Brand/content findings are routed to `brand-positioning` and
      `docs/brand/`.
- [ ] Accessibility-adjacent visual issues routed to `accessibility-review`.
- [ ] Functional failures routed to `functional-qa`.
- [ ] Repeated workflow lessons are routed through `closeout` or
      `codex-maintenance` to the narrowest memory owner.
- [ ] `docs-impact-map` is used when a visual finding may affect sibling
      product, design, brand, spec, backlog, glossary, or pattern docs.
- [ ] Evidence avoids sensitive data.
